# AGENTS.md вЂ” AI Agent Workflows for Trust-Bound

> **Purpose:** This file defines the autonomous and semi-autonomous AI agent workflows that assist both users and the development team. It is written to be read by both human developers and by AI coding assistants (Claude, Cursor, Gemini Code Assist) that will generate or extend this codebase.

## 1. What "Agents" Means in This Project

Trust-Bound uses two categories of agents:

**Runtime Agents (user-facing):** These are Gemini-powered AI features embedded in the product itself вЂ” the milestone generator, the contract drafter, and the dispute summarizer. They run in response to user actions and are fully documented in GEMINI.md.

**Development Agents (builder-facing):** These are AI coding assistants (Claude Sonnet, Cursor, or Copilot) that are given this AGENTS.md file as context to understand the project architecture and generate code that fits. This file is their primary briefing document.

This document primarily addresses the second category, but Section 5 covers runtime agent architecture in depth.

## 2. Project Context for Development Agents

If you are an AI coding agent reading this file, here is everything you need to know to generate correct, consistent code for Trust-Bound:

**Language:** TypeScript everywhere вЂ” both `backend/` (Node.js + Express) and `frontend/` (React 18 + Vite). Never generate plain JavaScript files in this project.

**Database access:** Always use the Prisma ORM client (`import { prisma } from "@/lib/prisma"`). Never write raw SQL queries except inside Prisma `$queryRaw` for performance-critical aggregations, and even then, always use tagged template literals (never string concatenation) to prevent SQL injection.

**API conventions:** All backend routes follow REST. Route files live in `backend/src/routes/`. Each route file exports a single Express Router. Route naming uses kebab-case (`/api/milestones`, `/api/dispute-chat`). All responses use the standard envelope: `{ success: boolean, data?: unknown, error?: string }`.

**Error handling:** Never throw unhandled errors. All async route handlers are wrapped with the `asyncHandler` utility from `backend/src/middleware/asyncHandler.ts`. Validation errors return HTTP 422 with a `fields` array. Auth errors return 401. Not-found returns 404. Server errors return 500 with a generic message (never expose stack traces in production).

**Frontend state:** Zustand is used for global state (auth session, wallet balance, active project). React Query (`@tanstack/react-query`) handles all server state (fetching, caching, mutation). Never use `useState` to cache server data вЂ” always use React Query.

**Component structure:** Components live in `frontend/src/components/`. Each component has its own folder containing `ComponentName.tsx`, `ComponentName.test.tsx`, and optionally `ComponentName.stories.tsx`. Components are never longer than 200 lines вЂ” split into subcomponents if they exceed this.

## 4. Agent Task Protocols

### 4.1 When Asked to Add a New API Endpoint

1. Check `SCHEMA.md` to understand the relevant database models.
2. Create the route in `backend/src/routes/[resource].ts`.
3. Create or update the controller in `backend/src/controllers/[resource]Controller.ts`.
4. Add Zod validation schema in `backend/src/types/[resource].schema.ts`.
5. Write the corresponding Jest test in `backend/tests/routes/[resource].test.ts` BEFORE or simultaneously with the implementation (TDD вЂ” see TDD.md).
6. Update the API client hook in `frontend/src/api/use[Resource].ts`.

### 4.2 When Asked to Add a New UI Component

1. Create the folder `frontend/src/components/[ComponentName]/`.
2. Build the component using Tailwind utility classes. Never write custom CSS unless Tailwind cannot achieve the desired result.
3. Use `shadcn/ui` primitives (Button, Dialog, Card, Badge, etc.) as the base layer вЂ” don't rebuild what already exists.
4. Write a test file using Vitest and React Testing Library.
5. Ensure the component is responsive (mobile-first) by default.

### 4.3 When Asked to Add an AI Feature

1. Read `GEMINI.md` completely before writing any Gemini-related code.
2. Place the Gemini service call in `backend/src/services/gemini/[featureName].ts`.
3. Never call the Gemini API directly from the frontend вЂ” always proxy through the backend to protect the API key.
4. Implement the fallback behavior described in GEMINI.md Section 6 before marking the feature complete.
5. Cache the AI response in the database if it is deterministic for a given input.

### 4.4 When Asked to Modify the Database Schema

1. Update `prisma/schema.prisma` first.
2. Run `npx prisma migrate dev --name describe_the_change` to generate the migration.
3. Update `SCHEMA.md` to reflect the change.
4. Check all controllers that query the changed model and update them.
5. Run the full test suite after migration to confirm nothing is broken.

### Agent State Management

Runtime agents are **stateless** вЂ” each call is independent. No conversation history is maintained between API calls. If a feature requires multi-turn interaction (e.g., the dispute summarizer reads a long chat), the entire relevant context is sent in a single API call, not across multiple turns.

This design choice avoids session complexity and makes the agents fully testable with deterministic inputs.

## 7. Notification Agent

A lightweight event-driven notification service sends real-time and email notifications on key escrow events. It listens to a PostgreSQL LISTEN/NOTIFY channel (or a simple in-process EventEmitter for the prototype) and dispatches:

- WebSocket push (via Socket.io) for real-time dashboard updates.
- In-app notification stored in the `notifications` table.
- Email notification via Nodemailer (using Gmail SMTP or Ethereal for testing).

Notifications are sent for: milestone submitted, milestone approved, dispute raised, dispute resolved, funds released, and invoice generated.

## 8. Instructions Specifically for Claude Code Agent

If you are Claude being used via Claude Code or a similar agentic coding environment, follow these additional instructions:

**Before writing any code:** Read `SCHEMA.md` for data models and `TDD.md` for the test conventions specific to this project. These two files will prevent the majority of integration errors.

**File creation:** Only create files in the directories defined in Section 3. Do not create helper files in the project root or add directories not in the structure.

**Imports:** Use path aliases (`@/lib/prisma`, `@/middleware/auth`, `@/types/project`) вЂ” these are configured in `tsconfig.json`. Never use deep relative imports (`../../../../lib/prisma`).

**Testing:** Write tests first, implementation second. If asked to add a feature and no test specification is provided, ask for clarification before generating implementation code.

**Comments:** Add a JSDoc comment to every exported function explaining its purpose, parameters, return value, and any side effects. This is especially important for wallet operations and Gemini service functions.

**Security:** Never log sensitive data (passwords, tokens, wallet amounts belonging to other users). Every route that modifies a resource must verify the authenticated user owns that resource. See `TDD.md` Section 4 for the authorization test matrix.
